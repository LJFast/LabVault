# LabVault

Personal lab value tracker. Local Obsidian vault with templates, schema, dashboards, and registry. Real lab data lives in a separate live vault on disk and is never committed to this repository.

## What's in this repo

```
LabVault/
├── README.md                     ← this file
├── .gitignore                    ← defensive — blocks any accidentally-added real data
├── v1/                           ← current vault skeleton
│   ├── 00_README.md              ← onboarding doc that goes inside the vault
│   ├── _meta/                    ← analyte registry, reference ranges, personal targets
│   ├── _templates/               ← Templater templates for new draws / imaging / TTE / reports
│   └── _dashboards/              ← Dataview-driven views (overview, trends, snapshot)
├── v2/                           ← future iterations land here
└── docs/
    ├── example_2026-04-28-lab.md ← real-shape example, identifiers removed
    └── design_notes.md           ← decisions and rationale
```

## What this repo is NOT

- It is not the live vault. The live vault holds your actual lab data and lives somewhere off this repo (e.g. `~/Documents/MyLabs/`).
- It is not a cloud-synced data store. No real values, no patient identifiers, no PDFs.
- It is not a clinical tool. It is a personal organisation system. Clinical interpretation requires your nephrology team.

## Setup (first time)

```bash
# 1. Choose a location for your LIVE vault, OUTSIDE this repo.
#    Example: ~/Documents/MyLabs/
mkdir -p ~/Documents/MyLabs

# 2. Copy the v1 skeleton into the live vault location.
cp -R v1/* ~/Documents/MyLabs/
cp -R v1/.* ~/Documents/MyLabs/ 2>/dev/null || true   # hidden files if any

# 3. Open ~/Documents/MyLabs/ as a vault in Obsidian:
#    Obsidian → "Open folder as vault" → select ~/Documents/MyLabs/

# 4. Install plugins (see v1/00_README.md for the list and config).

# 5. Verify dashboards render with the example draw note that ships
#    in docs/example_2026-04-28-lab.md — you can copy that into
#    ~/Documents/MyLabs/Labs/2026/ for testing, then delete before
#    adding real data.
```

## Maintenance and versioning

- **Structural changes** (new analytes, schema tweaks, new templates) → edit files under `v1/`, commit.
- **Big breaking change** (e.g., schema refactor that requires migrating existing notes) → create a new `v2/` folder, write a migration note in `docs/`, commit.
- **Don't touch the live vault** with Git operations. The live vault holds real data; treat it as a separate, untracked working store with its own backup strategy.

## Backup of the live vault

The live vault is just a folder of markdown files. Recommended:

- Time Machine (macOS) or equivalent local backup with encryption.
- Optional: an encrypted archive (e.g., 7z with password, age, or a VeraCrypt container) backed up to external drive.
- **Not recommended without thought:** iCloud Drive, Dropbox, OneDrive, GitHub, or anything else that puts the data on a third-party server.

## Privacy stance

The vault is designed so that:

1. Source PDFs stay outside the vault (kept locally, encrypted at rest).
2. Notes contain values, units, ranges, and dates — not patient identifiers.
3. AI-assisted interpretation uses anonymised snapshots generated by a Dataview query, copy-pasted into a fresh chat. The original notes are never uploaded.

This is documented in detail in `v1/00_README.md` and `v1/_dashboards/snapshot_for_claude.md`.
